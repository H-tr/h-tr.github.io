# Closed Domain Q&A system介绍
回顾之前所说的，我们知道一个问答系统的实现要分为两部分：document retriever和document reader。在动手写问答系统-5中，我们获得了一个非常好的document reader，能够将长文本拆分成短文本并放入语言模型中进行训练。在这里我们将直接应用这个document reader而不是重新训练一个。

现在我们做欠缺的就是一个类似wiki engine的文档搜索引擎。我们将会自己通过语言的相似程度自己实现一个。
## 怎么做
想要一个文档搜索引擎，我们到底应该实现些什么？

### 数据集
首先我们应该有一个数据集，这个数据集就是QAsystem的知识来源。由于大多数的文档都是pdf格式，我们在这里就使用一套pdf的文档作为我们的数据集。在这里我使用的是machine learning的一些经典文章：
- AlexNet
- GAN
- PGT3
- MAE
- RestNet
- Transformer
- ViT

如果大家想要更好的了解这些文章讲的东西，除了直接阅读之外，也可以去bilibili听李沐老师讲论文。

使用论文的好处就是它们都集中在一个主题上，而且每一篇论文都有自己的独到之处。既方便测试搜索引擎的所搜能力，也方便检验语言模型的提取能力。

### 知识图谱(Knowledge Graph)
知识图谱是一种使用“图”的拓扑结构来存储知识的数据库。我们一般来说不太容易直接从一堆文档中搜索到我们想要的，所以应该先用这些文档建立起来一个数据库，然后从数据库中搜索文档并进行语义解析。

### 语义解析(Semantic Parsing)
之前我们建立open domain Q&A system的时候直接用的wikipedia的搜索引擎来寻找文档，wiki engine直接帮我们完成了解析语义的步骤。但是当我们自己建立数据库的时候，就需要自己建立语义解析机制。这个可以直接通过传统的逻辑分析完成，也可以用nlp模型完成。解析完问题的语义之后才可以从数据库里搜寻相关性最大的文档。

### 准备工作
因为我们手上的数据全都是pdf的形式，我们需要把它们转成语言模型能够处理形式，然后喂给语言模型。

在这篇文章之前，我们使用GPT3和BERT两种语言模型都简单的实现了问答系统，并获得了还不错的效果。鉴于两者都可以实现，我们只需要选择其中一种就可以了。这篇文章选的是BERT，无它，免费尔。

我们之前使用的GPT3实现的问答系统，实际上是通过语言补全的功能把预测做成了问答系统的样子。它对于大段文字的理解能力有限，更多的是依赖于自己预训练时得到的知识，然后对对话进行补全。这样一来如果我们想要给它安排一些先验的知识就会比较困难。想要做成一个比较完善的问答系统，需要GPT3专门的问答功能。但是这个功能是收费的。作者贫穷，这个项目的老板也不想额外掏钱，所以我们就决定不使用GPT3。

但是GPT3是已经实现了非常完整的api的。想要开发的话只需要调用api，然后把文档传给它的server，他们就会自动帮你完成文档搜索和语义理解，非常的方便。我们给出一个参考代码，但是没有实际运行过，如果有问题希望能够指正：
```python
class DocumentReader:
    def __init__(self) -> None:
        pass

    def chat(self, prompt, engine='text-davinci-002', response_length=64,
            temperature=0.7, top_p=1, frequency_penalty=0, presence_penalty=0,
            start_text='', restart_text='', stop_seq=[]):
        response = openai.Completion.create(
            prompt=prompt + start_text,
            engine=engine,
            max_tokens=response_length,
            temperature=temperature,
            top_p=top_p,
            frequency_penalty=frequency_penalty,
            presence_penalty=presence_penalty,
            stop=stop_seq,
        )
        answer = response.choices[0]['text']
        new_prompt = prompt + start_text + answer + restart_text
        return answer, new_prompt

    def create_answer(self, question, doc_list, examples_context="Larry Page and Sergey Brin.", 
        search_model="davinci", model="curie") -> String:
        response = openai.Answer.create(
            search_model = search_model,
            model = model,
            question = question,
            documents = doc_list,
            examples_context = examples_context,
            examples=[["Who founded Google?"]],
            max_tokens=30,
            stop=["\n", "<|endoftext|>"]
        )
        answer = response["answers"][0]
        return answer
```

然后我们来进行一下问答系统所需要的预处理。第一件事情就是把pdf转换成文本格式：
```python
from io import StringIO
from pdfminer.pdfinterp import PDFResourceManager, PDFPageInterpreter
from pdfminer.converter import TextConverter
from pdfminer.layout import LAParams
from pdfminer.pdfpage import PDFPage

def convert_pdf_2_text(path):

    rsrcmgr = PDFResourceManager()
    retstr = StringIO()

    device = TextConverter(rsrcmgr, retstr, codec='utf-8', laparams=LAParams())
    interpreter = PDFPageInterpreter(rsrcmgr, device)

    with open(path, 'rb') as fp:
        for page in PDFPage.get_pages(fp, set()):
            interpreter.process_page(page)
        text = retstr.getvalue()

    device.close()
    retstr.close()

    return text
```

像模型提出问题，比如说：
> How large is the patch of resulting image?

这个问题问的非常含混。如果我们对general QA system提出这个问题的话，它一定不知道怎么回答，或者说会给出一个错误的答案。毕竟没有任何先验知识的话即使是给一个人提出这个问题，人也是不知道应该怎么直接回答的。但是模型给出了答案：
```
BERT: 224 × 224 × 3
```
这确实是AlexNet中的patch size。是一个三维张量，每个图片224*224像素，每个像素有rgb三个通道。这说明我们的模型确实是从文章中获得了先验的知识，他知道我们提的问题是针对这个文章的。pdf的解析也非常成功，能够让模型有效的获得文档里的知识。

但是如果只有函数的话在调用时太不方便了，而且拓展起来也会很费劲。所以我们把它改写成class：

这里的initialize部分是读取所有的pdf，然后把他们都存作csv的格式。这样能够方便后续的处理和调用，再次读取的时候也省时间，不需要重新处理一遍。
